Finding Leaks With Gdb

Suppose you have in your software objects with specific lifetime requirements, e.g. they must be created during initialization or they must not persist beyond one cycle. Moreover, a violation of these requirements may lead to a crash and you know which object caused the problem but you do not know where it came from.

Debugging might be very difficult if your software is sufficiently large and I’d like to share what I did in this situation. I had

Object* factory() { ... }

which was called in several dozen places throughout the code. I was tempted to store __FILE__ and __LINE__ with the object but modifying every call to factory() was too tedious.

Fortunately, it is possible to find the file name and line number of each call to factory() from inside the function body.

Enter __builtin_return_address

As the name implies, this is a builtin function in gcc (and clang), which takes an unsigned int argument (the level, but we will ignore this) and returns a void pointer. Per gcc’s documentation:

This function returns the return address of the current function […]

In order to better understand what it does, let’s look at the assembly output of this very simple program:

void* foo() {
	return __builtin_return_address(0);
}

int main() {
	foo();
}

If we compile1 it and run it with gdb, we get

0x5555555545fa <foo()>                  push   rbp
0x5555555545fb <foo()+1>                mov    rbp,rsp 
0x5555555545fe <foo()+4>                mov    rax,QWORD PTR [rbp+0x8]
0x555555554602 <foo()+8>                pop    rbp
0x555555554603 <foo()+9>                ret
0x555555554604 <main()>                 push   rbp
0x555555554605 <main()+1>               mov    rbp,rsp
0x555555554608 <main()+4>               call   0x5555555545fa <foo()>
0x55555555460d <main()+9>               mov    eax,0x0
0x555555554612 <main()+14>              pop    rbp
0x555555554613 <main()+15>              ret

The interesting bit happens at foo()+4, where the value of [rbp+8] is moved into rax2. In gdb we can inspect this value and see

(gdb) p/x *(unsigned long*)($rbp + 8)
$1 = 0x55555555460d

which is main()+9, the code immediately after the call to foo(). What we just did manually, gdb can do automatically:

(gdb) info line *0x55555555460d
Line 7 of "research.cpp" starts at address 0x55555555460d <main()+9> and ends at 0x555555554614.

Note that gdb tells us file name and line number.

Finding leaks

Returning to our example before, I modified Object and the factory

Object* factory() {
	Object* ret = ...
	ret->m_origin = __builtin_return_address(0);
	return ret;
}

When the bug occured and I knew which object caused the issue, I was now able to figure out where it came from:

(gdb) info line *offendingObject->m_origin

This technique often allowed me to find the needle in the haystack, for example where an exception was thrown, with little time investment.
